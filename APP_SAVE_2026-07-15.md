## Features Currently Built

### Student Experience

- Main menu with Learn, Quiz, Refresh, Match, Insights, Blitz, and Leaderboard.
- Flashcard quizzes and written-answer flow.
- Active Assignments widgets on dashboard, quiz, written quiz, and blitz views.
- Assignment loading routes students directly into the required cards or long-answer question.
- Assignment completion requires reaching the teacher target mastery percentage.
- XP uses active engagements rather than raw stopwatch time.
- Student rank display and top-three leaderboard badges.
- Student nudge inbox with read acknowledgement.
- Flag Error flow writes reports to `flagged_content`.

### Teacher Experience

- Educator Command Center with classes, student counts, subject access, and active assignment completion.
- Multi-class teacher schema.
- Class display-name editing.
- License and seat management UI for licensed teachers/admins.
- Assignment Engine supports:
  - whole chapter assignments,
  - subsection assignments,
  - long-answer assignments,
  - target mastery percentage,
  - deadline editing,
  - canceling active assignments.
- Student Progress Overview table is collapsible and horizontally scrollable.
- Teachers can view/select students and inspect mastery breakdowns.
- Teachers can nudge a selected student or nudge all students with incomplete assignments.

### Super Admin Experience

- Hidden admin control panel behind `REACT_APP_SUPER_ADMIN_KEY`.
- Admin can preview clean student and teacher layouts without a simulation.
- Admin Simulation Lab can generate random schools with teachers, classes, and students.
- Simulation supports:
  - multiple classes and teachers,
  - teacher with multiple classes,
  - 1 to 50 simulated days,
  - real-time-style speed multiplier,
  - persistent play/pause/speed dock across simulation views,
  - live student telemetry,
  - collapsible telemetry table,
  - manual nudges/rewards,
  - automatic idle nudges and streak rewards,
  - CSV-style data table export for analysis.

### Curriculum / White-Label Foundation

- Firestore-backed `curriculums/{subjectId}` collection.
- Legacy `data.js` remains as fallback and seed data.
- Admin Curriculum Editor can edit question text, answers, mark schemes, marks, and image URLs.
- Immutable question IDs are preserved so historical mastery keys remain stable.
- Bulk curriculum import path exists through the admin curriculum area.
- Firestore-backed `licenses/{licenseId}` collection supports schools, unlocked subjects, class limits, and seat limits.

## Firestore Collections In Use

- `users/{email}`: user profile, role, class membership, streak, XP, written progress.
- `users/{email}/progress/{cardId}`: per-card mastery/progress.
- `assignments/{assignmentId}`: teacher-created assignments and completion state.
- `nudges/{nudgeId}`: teacher/admin messages to students.
- `curriculums/{subjectId}`: subject curriculum data.
- `flagged_content/{flagId}`: student content error reports.
- `licenses/{licenseId}`: school/license/seat/class entitlement data.

## Known Unfinished Tasks / Next Checks

1. Student leaderboard access needs a rules/design review.
   The UI ranks classmates from `users`, but the current rules only let a student read their own user document while teachers/admins can read all users. A safer future design is a class-scoped public leaderboard/profile mirror or a rules update that only exposes classmates in the same class.

2. Assignment completion rules should be hardened.
   Current rules allow signed-in users to update `completedBy` and `updatedAt` on assignments. This should be restricted so a student can only mark their own completion and only for assignments linked to their class.

3. Bulk curriculum import needs real dataset QA.
   The import foundation exists, but the final production workflow should be tested with a full AQA/real exam-board dataset, including chapter/subsection IDs, long-answer questions, mark schemes, image URLs, and version naming.

4. License onboarding is still manual.
   License documents can be used by the app, but school purchase, IT admin provisioning, teacher invites, and seat allocation are not a polished self-serve onboarding flow yet.

5. Visual QA still needs device testing.
   Tables were improved for laptop/mobile behavior and the build passes, but real iPhone, Android, Mac, Windows, and tablet screenshots should still be checked before a school demo.

6. Admin simulation is intentionally local-state only.
   This keeps production metrics clean. If persistent simulation scenarios are ever needed, they should go into a separate sandbox collection/project, not production student metrics.

7. Firestore data seeding/admin bootstrapping is still partly manual.
   The root admin user, licenses, and initial curriculum seed may need manual creation or a future safe admin-only seed script.

8. Browser preview from Codex was blocked by sandbox port permissions.
   Use Vercel preview/production or a normal local terminal to visually test the app.
