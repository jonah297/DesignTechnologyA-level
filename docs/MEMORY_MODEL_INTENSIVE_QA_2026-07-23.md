# Sharp Study Memory Model Intensive QA

Date: 23 July 2026

Purpose: stress test the `sharp-dsr-1` memory model across a wider, more realistic set of student behaviours before using it as the basis for teacher reports, student dashboards, refresh recommendations, and simulation tooling.

This is a deterministic local algorithm test. It does not write to Firebase, create accounts, alter assignments, change XP, or affect production leaderboards. The outputs below are generated by running the actual memory model functions through simulated study behaviour. They are not used as fixed test fixtures.

## How To Rerun

```bash
npm test -- --watchAll=false src/memoryModelSimulation.test.js
```

To print the cohort summary table:

```bash
MEMORY_INTENSIVE_REPORT=1 npm test -- --watchAll=false src/memoryModelSimulation.test.js --silent=false
```

Relevant files:

- `src/memoryModel.js`
- `src/memoryModelSimulation.js`
- `src/memoryModelSimulation.test.js`

## Test Method

The intensive cohort runs six student profiles over a 365-day academic-style period using 96 simulated cards:

| Profile | Behaviour being tested |
| --- | --- |
| Consistent high performer | Studies on most weekdays, learns early, repairs weak cards quickly. |
| Steady average student | Studies three or four times per week with gradually improving accuracy. |
| Struggling but consistent | Shows up often but starts with low accuracy. Should improve, but not be treated as perfect. |
| Inconsistent student | Attempts work occasionally, has quiet spells, then responds to catch-up/nudge windows. |
| Deadline crammer | Does little for long periods, then works hard around deadline windows. |
| Absent then repairs | Builds early memory, disappears for a month, then returns with repair sessions. |

The tests check shape and behaviour, not exact memorised output:

- all metrics remain finite and bounded from day 0 to day 365;
- consistent study beats deadline cramming;
- absence increases refresh pressure;
- repair sessions reduce refresh pressure;
- struggling students can recover, but remain behind stronger learners;
- partial course coverage remains visible in whole-topic mastery;
- a fake injected memory model changes the output, proving the report is not a hard-coded table.

## Findings

1. Consistency is rewarded. The high performer finishes with the highest mastery, lowest final refresh pressure, and strongest stability.

2. Deadline cramming can recover visible marks but creates weaker memory. The crammer reaches 88.7% final mastery, but with much lower stability than the consistent learner.

3. Absence creates meaningful decay. The absent-then-repair profile drops during the quiet period and recovers when repair sessions resume.

4. The inconsistent student now behaves realistically. They learn part of the course, build some mastery, remain behind on coverage, and carry a larger final refresh load.

5. Whole-topic mastery must stay separate from learned-card mastery. The struggling profile has 90.9% learned-card mastery but only 75.7% whole-topic mastery because 16 cards were never learned. Teacher dashboards should use whole-topic mastery for top-line reporting, with learned-card mastery shown only as detail.

6. Low total accuracy plus heavy repetition can still produce high learned-card mastery. This is not automatically wrong, because repeated recall can repair memory, but it means the UI should show accuracy, coverage, and refresh load beside mastery.

## QA Verdict

The memory model is strong enough to keep developing with. It passes the important behavioural checks:

- students do not collapse after one missed day;
- missed time creates refresh demand;
- repair sessions work;
- consistent practice beats last-minute cramming;
- partial coverage is visible;
- fake model injection changes output, so the test is genuinely exercising the algorithm.

The next tuning work should focus on presentation and thresholds rather than replacing the model:

- teacher dashboard top-line mastery should use coverage-weighted whole-topic mastery;
- student detail views should show learned-card mastery, coverage, refresh load, and recent accuracy together;
- nudge wording should target the reason: overdue assignment, high refresh load, low coverage, or streak risk;
- future school-pilot reports should avoid using one mastery number on its own.
